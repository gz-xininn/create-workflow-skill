## What is this?

`create-workflow-skill` is a Claude Code skill that generates a complete multi-role team collaboration workflow environment. It creates the full directory structure, role system, guardrail rules, skill templates, and knowledge base — enabling you to quickly build an AI-driven development collaboration process.

### Key Features

- **Multi-role System** — PM, Frontend, Backend, QA, DevOps, Acceptance, with RBAC permission matrix
- **Task Workflow** — Semantic understanding → Task initialization → Execution → Context handoff
- **Security Guardrails** — Database, Git, data, and execution red-line rules that cannot be bypassed
- **Knowledge Base** — Context compression & handoff, saving 70-80% tokens across sessions
- **Built-in Skills** — 6 ready-to-use skills: task-init, context-handoff, knowledge-sync, acceptance-check, security-gate, generate-help
- **Fully Customizable** — Add custom roles, skills, and prompt templates

## Quick Start

### 1. Install the Skill

Copy the `SKILL.en.md` file and `templates/` directory to your Claude Code skills directory:

```bash
# Create the skill directory
mkdir -p .claude/skills/create-workflow-skill

# Copy the skill file (rename to SKILL.md) and templates
cp SKILL.en.md .claude/skills/create-workflow-skill/SKILL.md
cp -r templates .claude/skills/create-workflow-skill/templates
```

Or clone this repository directly:

```bash
git clone https://github.com/gz-xininn/create-workflow-skill.git
cp create-workflow-skill/SKILL.en.md /your-project/.claude/skills/create-workflow-skill/SKILL.md
cp -r create-workflow-skill/templates /your-project/.claude/skills/create-workflow-skill/templates
```

> **Note**: For the Chinese version, copy `SKILL.md` instead of `SKILL.en.md`:
> ```bash
> cp SKILL.md .claude/skills/create-workflow-skill/SKILL.md
> ```

### 2. Run the Skill

In Claude Code, type:

```
/create-workflow-skill
```

Or use natural language:

```
init workflow
```

### 3. Follow the Wizard

The skill will guide you through:

1. **Project info** — Enter project name and root directory
2. **Select roles** — Choose which roles you need (default: all 6)
3. **Select skills** — Choose which skills to include (default: all 6)
4. **Generate** — All files are created automatically
5. **Customize** — Fill in your project-specific details in CLAUDE.md

## Task Workflow

Every task follows this mandatory process:

```
1. Semantic Understanding    →  Parse user input, output understanding template

2. Task Initialization       →  Generate structured task sheet via /task-init

3. Execution                 →  Work within confirmed scope, follow guardrails

4. Context Handoff           →  Compress & save via /context-handoff
```

### Development Workflows

**Feature Development:**
```
PM (Requirements) → FE/BE (Design & Code) → QA (Test) → OPS (Deploy) → ACC (Accept)
```

**Hotfix:**
```
Issue confirmed → FE/BE (Minimal fix) → QA (Quick verify) → OPS (Emergency deploy)
```

**Release:**
```
Release branch → QA (Full regression) → Bugfix only → ACC (Sign-off) → OPS (Deploy & tag)
```

## Security Guardrails

The workflow enforces strict security rules that **cannot be bypassed**:

### Red Lines

| Category | Key Rules |
|----------|-----------|
| **Database** | No production DB access; No DELETE/DROP/TRUNCATE; Soft-delete only; Migration requires approval |
| **Git** | No force-push to protected branches; No --no-verify; Conventional Commits required |
| **Data** | No production secrets/tokens in output; No real PII exposure; Code shared with AI must be sanitized |
| **Execution** | No destructive commands without confirmation; No dependency changes without approval; Scope cannot expand without re-running /task-init |

### Execution Boundaries

| Task Type | Allowed | Prohibited |
|-----------|---------|------------|
| Feature | Create/modify in scope, add tests | Modify outside scope, change infrastructure |
| BugFix | Fix defect files, add regression tests | Refactor, change unrelated code |
| Refactor | Rename, extract, restructure | Change behavior, modify API contracts |
| Analysis | Read-only, generate reports | Modify any files |
| Deploy | Deployment scripts & configs | Modify source code |

## Knowledge Base

The knowledge base is the **core token-saving mechanism**:

- **Context compression**: 200-400 tokens replace 3000-5000 tokens of conversation replay (70-80% savings)
- **Cross-session continuity**: Task context persists across Claude Code sessions
- **Cross-role handoff**: PM context flows seamlessly to Dev, QA, DevOps, and Acceptance

### Structure

```
knowledge-base/
├── INDEX.md                    # Master index (always check first)
├── contexts/
│   ├── active/                 # In-progress task contexts
│   └── archived/               # Completed task contexts
├── prompts/                    # Reusable prompt templates
│   ├── pm/                     # PM templates (PRD, user stories, etc.)
│   ├── frontend/               # Frontend templates
│   ├── backend/                # Backend templates (API design, etc.)
│   ├── qa/                     # QA templates (test plans, bug reports)
│   ├── devops/                 # DevOps templates (deploy checklists)
│   └── acceptance/             # Acceptance templates
└── decisions/                  # Architecture Decision Records (ADR)
```

## Customization

### Add a Custom Role

1. Create `docs/rules/roles/{role-id}.md` following the role template
2. Add the role to `ROLES.md` (role matrix, permission matrix, skill mapping)
3. Update `CLAUDE.md` role routing section

### Add a Custom Skill

1. Create `.claude/skills/{skill-name}/SKILL.md` with frontmatter
2. Add trigger words in the `description` field
3. Register in `CLAUDE.md` available skills table

### Add Prompt Templates

Use `/knowledge-sync` or manually create files in `knowledge-base/prompts/{role}/` with:

```markdown
---
name: template-name
role: role-id
version: 1.0
last-updated: YYYY-MM-DD
description: One-line description
---

# Template Title

## Applicable Scenarios
When to use this template

## Template Content
Actual prompt template with {placeholders}

## Usage Example
A filled-in example
```

## Design Philosophy

### Why This Workflow?

1. **Anti-hallucination** — Mandatory semantic understanding step prevents AI from guessing or fabricating requirements
2. **Scope control** — Task boundaries prevent scope creep; every expansion requires re-confirmation
3. **Security first** — Red-line rules cannot be bypassed, even with explicit user requests
4. **Token efficiency** — Knowledge base compression saves 70-80% context tokens across sessions
5. **Role isolation** — RBAC prevents PM from touching code, prevents Dev from deploying, etc.
6. **Traceable** — Every decision, change, and handoff is recorded and auditable

### Architecture Principles

- **Convention over configuration** — Sensible defaults, customize only what differs
- **Structure and content separation** — The skill generates the framework; users fill in project-specific content
- **Claude Code native** — Built entirely on Claude Code's native capabilities (rules, skills, CLAUDE.md)
- **No external dependencies** — Pure markdown files, no build tools, no runtime dependencies

## FAQ

**Q: Does this work with any tech stack?**
A: Yes. The workflow is tech-stack agnostic. It generates a role-based collaboration framework — you fill in your specific tech stack in CLAUDE.md.

**Q: Can I use fewer roles?**
A: Absolutely. During setup, deselect any roles you don't need. A solo developer might only use BE + QA, for example.

**Q: Does this require Claude Code Pro?**
A: This skill works with any Claude Code plan that supports custom skills.

**Q: How do I update after generation?**
A: Edit any generated file directly. Use `/knowledge-sync` for knowledge base updates. Re-run `/create-workflow-skill` to regenerate (it will show diffs and ask before overwriting).

**Q: Can multiple people use this in the same repo?**
A: Yes. The role system is per-session — each Claude Code session picks a role. The knowledge base (contexts, decisions) is shared for cross-role collaboration.

**Q: Does this workflow only support Chinese?**
A: No. You can choose Chinese, English, or bilingual mode in the setup wizard.
